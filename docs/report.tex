\section{Introduction}
This report outlines the implementation and analysis of a C/C++ subroutine designed to compute the $k$-Nearest Neighbors ($k$-NN) for a query set $Q$ of $n_Q$ points against a corpus set $C$ of $n_C$ points, both in $d$-dimensional space.
For this project, an exact distance algorithm was implemented in serial as a baseline (Version 0), and then parallelized using two main frameworks: OpenMP and POSIX Threads (Pthreads) as Version 1.

\section{Mathematical Method \& Distance Calculation}
Instead of generating a full $O(n^2)$ Euclidean distance matrix, which would quickly exhaust memory for large inputs, the algorithm operates by evaluating the expanded distance formula:
\begin{equation}
D(C, Q) = \sqrt{C^2 - 2 C Q^T + Q^2}
\end{equation}

In this setup, each query point (or column) of $Q$ is processed iteratively. To maximize calculation efficiency, the $-2 C Q^T$ term is computed using \texttt{OpenBLAS} as specified by the assignment. Specifically, the highly optimized level-2 BLAS function \texttt{cblas\_dgemv} was used to handle the dense matrix-vector multiplications.

Following the matrix computations, an $O(N)$ QuickSelect partitioning algorithm isolates the exact top $k$ minimum distances. This provides the nearest neighbors without the $O(N \log N)$ computational overhead of sorting the entire distance array.

\section{Parallelization Methodologies}
When testing symmetric queries over identically sized inputs ($C=Q$), the independent distance evaluations can be parallelized horizontally.

\subsection{OpenMP Implementation}
OpenMP simplifies parallelization by using compiler pragmas to handle threaded execution logic. By wrapping the main query loops with \texttt{\#pragma omp parallel} and \texttt{\#pragma omp for}, OpenMP dynamically maps subsets of the query execution across available cores. Furthermore, thread contexts allocate memory frames per loop chunk (preventing race conditions) and seamlessly handle thread joining upon completion.

\subsection{Pthreads Implementation}
With Pthreads, thread management is handled manually. The query set $Q$ is divided into equal chunks (using integer slicing), and a struct mapped to the custom \texttt{knn\_worker} routine passes the start and end indices to each thread. Threads are explicitly created with \texttt{pthread\_create()}, process their assigned bounds, and integrate using \texttt{pthread\_join()}. To prevent deadlocks from the nested OpenBLAS computations attempting to spawn their own threads on top of our OS threads, we constrain OpenBLAS to a single thread per worker using \texttt{openblas\_set\_num\_threads(1)}.

\section{Correctness Checks \& Validation}
To verify correctness during development, the algorithm's output was first tested using simplistic datasets. Then, to confirm consistency formally, a Python script was implemented (\texttt{utils/validate.py}) to generate symmetric pseudo-random datasets and feed them to the compiled C binaries. Subsequently, their output is compared against \texttt{scikit-learn}'s \texttt{NearestNeighbors(algorithm='brute')}. Finally, the outputs are validated globally by ensuring the maximum absolute difference between the distance matrices is within an acceptable floating-point tolerance (\texttt{np.max(np.abs(dst\_c - dst\_py)) < 1e-4}), guaranteeing that our C implementation demonstrates parity with the Python reference.

\section{Calculation Performance \& Speedups}
Experiments were run against uniformly generated datasets (a $10,000 \times 10$ Corpus against $1,000$ Queries, evaluating $k=5$) as per the defaults of the \texttt{validate.py} script. On the same basis, performance was  benchmarked by dedicating exactly 4 threads to the parallel executions.

\subsection{Speedup Evaluation}
\begin{itemize}
    \item \textbf{Version 0 (Serial Base)}: The baseline execution time was approximately $0.78$ seconds, relying strictly on OpenBLAS.
    \item \textbf{Version 1 (OpenMP)}: Utilizing 4 threads, execution dropped to $\approx 0.05$ seconds (a $\approx 15.46\times$ speedup over the baseline). This exceptional scaling suggests OpenMP efficiently manages thread overhead while allowing OpenBLAS to fully utilize the underlying hardware.
    \item \textbf{Version 1 (Pthreads)}: The Pthreads implementation executed in $\approx 0.126$ seconds (a $\approx 6.19\times$ speedup). While significantly faster than the serial version, manual thread chunking introduces more overhead. However, limiting OpenBLAS to 1 thread per worker successfully prevented OS starvation, allowing the manual threading to scale reliably across the cores.
\end{itemize}

Overall, these metrics clearly demonstrate the massive performance gains achievable when combining optimized math libraries like OpenBLAS with careful multi-core parallelization.
